## Короткий Вывод

- Старый parity score по `v0.12.0` больше не считать главным ориентиром: `codex-acp-upstream v0.16.0` уже подтянул `agent-client-protocol = 0.14.0` и `codex rust-v0.137.0`, ACP `v0.13.5`/`v0.13.6` стабилизировал `additionalDirectories`, session usage updates и `session/delete`, а `v0.14.0` главным образом оформил Rust schema crate в отдельный workspace package.
- Основные пробелы относительно свежего upstream/release-notes направления теперь: full `additionalDirectories` runtime bridge, полная MCP elicitation parity, richer account usage/rate-limit/reset-credit surface, новый `codex` app-server protocol drift по collab/subagents и отдельные client-native UX-ветки, которые зависят от свежего Zed ACP UI. Нативный ACP `auth.logout`/`logout` в форке поддержан с `0.24.0`, а ACP `session/delete` для Zed history delete - с `0.26.0`.
- Основные сильные стороны форка: отдельный `resume_session`, workspace-scoped `/resume`, `/threads`, `/plan`, app-server-ориентированный flow восстановления тредов, нативный Zed context donut через ACP `usage_update`, `Speed` service-tier selector, более точные ACP `ToolCallLocation` для Zed Follow и отдельный режим ручного restore через `ACP_DISABLE_AUTO_RESTORE=1` + `/resume`.
- Свежий Zed стал заметно сильнее вокруг внешних агентов: registry-first install path, Threads Sidebar / Parallel Agents, session history/restoration для external agents, running agent version в External Agent settings, editable external-agent thread titles, logout UI, additional directories, subagent loading/rendering and permission surfacing, ACP registry download indicator, и лучшие startup errors. Но docs Zed всё еще явно фиксируют, что для external agents часть возможностей может отсутствовать, а queued messages у external agents уходят только в конце generation, не на tool-call boundary.
- Отдельного нового ACP `plugin agent` / `subagent` wire-типа в свежем срезе не найдено. Ближайший upstream-направляющий документ — ACP RFD про proxy chains: расширения/skills/MCP/subagents предлагается моделировать через proxy/conductor + MCP-over-ACP, а не через новый основной `ThreadItem`-тип. Для форка это означает: не добавлять adapter-side "plugin agent" runtime-ветку без конкретного клиента/протокола; сначала документировать нужный proxy/ACP контракт.

1. Не поднимать `agent-client-protocol` и `codex` одновременно без отдельного compile spike; ACP SDK уже поднят отдельно в `0.26.0`.

## Правило Для Новых UX-Фич

Перед вводом новой user-facing фичи сначала проверять ownership: закрывается ли она кодом этого адаптера, или зависит от `Zed`, `ACP`, `codex app-server` либо другого upstream-контракта. Если видимый UI рисует `Zed` и ACP не дает явного поля/метода для управления этим UI, фича считается Zed-side до доказательства обратного. В таком случае в runtime адаптера не добавляется "готовая" ветка ради теоретического parity; сначала документируется ограничение, нужный Zed-side patch/контракт и fallback, который реально доступен в форке.

### Правило Для GitHub Releases / Release Notes

При обновлении references и сравнении «что нового» сначала читать GitHub Releases / release notes / changelog по каждому upstream, а не начинать с code diff. Для Zed external-agent работы фильтровать строки по `ACP`, `external agents`, `agent_servers`, `ACP Registry`, `CustomAgentServer`, `Codex`, `Claude Agent`, `Gemini CLI` и `OpenCode`. Общие `Agent:` пункты учитывать только если они явно затрагивают ACP, external-agent sessions/history, permissions, subagents или settings для внешних агентов. Нативные Zed Agent provider/model/tool UX-пункты не считать задачами адаптера, пока код или протокол не показывают, что они реально проходят через ACP.

Каждый кандидат из release notes проверять точечно в соответствующем `references/*` коде: для Zed обычно `references/zed/crates/agent_servers`, `references/zed/crates/acp_thread` и `references/zed/crates/agent_ui`; для ACP/Codex - только нужные protocol/schema/app-server участки. После проверки отдельно фиксировать ownership: Zed-side only, новый/измененный ACP контракт, app-server protocol drift, или adapter-side mapping. Client-side фичи вроде editable external-agent thread titles считать локальными Zed metadata (`title_override`), пока Zed не отправляет явный ACP request/event назад в агент; backend-persistent Codex title остается путем `/rename` -> `thread/name/set`.

## 4. Что Стоит Подумать Всерьез

На текущем этапе для форка под `Zed` разумно держать такой shortlist:

1. Развести две миграции: сначала bounded bump `codex` crates к осознанно выбранному свежему commit, затем отдельно ACP SDK bump. `agent-client-protocol v0.13.6` уже содержит v1/v2 split, MCP-over-ACP scaffolding, stabilized `logout`, `additionalDirectories`, session usage updates, `session/delete` and unstable plan operations, поэтому слепой подъем вместе с Codex protocol drift слишком рискованный.
2. Спайкнуть `additionalDirectories` как первый bounded ACP parity patch: свежий Zed уже умеет ACP additional directories, ACP это стабилизировал, а текущий форк пока живет вокруг single `cwd` + app-server thread state.
3. Перенести маленькие high-value parity patches из `codex-acp-upstream v0.16.0`: richer permission/account-limit mapping, plan update rendering, session usage/delete semantics и точечные MCP elicitation детали. Image-generation уже surfaced в форке как прагматичная adapter-side ветка, а ACP `logout` уже закрыт.
4. После dependency bump отдельно прогнать audit approval-flow: command/file/user-input permissions, `RequestPermissions`, MCP elicitation approvals, strict auto-review. Это зона с самым высоким риском тихих семантических регрессий.
5. Поднять `Auto-review` как отдельный guarded permission mode, а не как bypass: selector item рядом с `Workspace`, reviewer policy `auto_review`, lifecycle UI для `item/autoApprovalReview/*`, явное отличие от `Full access`.
6. Для новых slash/preview UX сначала требовать явный пользовательский сценарий. `thread/read` остается recovery-механизмом для восстановления resume-path, но не surfaced preview-flow сам по себе.

Отдельное UX-направление, которое стоит держать рядом с этим shortlist:

- Канонический status surface теперь разделен: нижний `Status` selector держит быстрые read-only пункты `Limits`, `Session`, `MCP`, `Skills`, `Plugins` и action `Compact now`, а `/status` выводит полный чатовый отчет `Limits` / `Session` / `Integrations`. Context usage больше не дублируется текстовым selector-ом: свежий Zed уже рисует нативный context donut для custom ACP CAS из нашего ACP `usage_update`.
- Следующий вопрос теперь не “как ещё назвать status-команду”, а какие read-only summaries реально нужны пользователю рядом с уже существующим `/status`, `/diff` и selector UX. Не добавлять новые preview-flow только потому, что transport уже есть.
- Для нового чистого чата канонический путь теперь native `Zed` `New Thread`. In-place switch в рантайме сознательно оставлен только для `/fork` и archive-triggered replacement; пока сам `Zed` не научится reset'ить ACP session view, старые сообщения в sidebar останутся видимыми после таких сценариев.
- Для свежего Zed канонический framing теперь шире одного Agent Panel: Threads Sidebar / Parallel Agents позволяют держать несколько external agent threads рядом с native Zed Agent, а worktree isolation стало штатным Zed-side решением для параллельных изменений. Адаптеру не надо имитировать это runtime-командами, если Zed уже дает native оболочку.

### 4.1 UX Backlog

Идеи ниже зафиксированы как practical follow-ups после сжатия нижней панели selectors и не требуют Zed-side патча, если не указано обратное:

- `display-maps`: держать только account-limit variants для `Status`, `/status` и warning text. Context percent/braille styles удалены; context usage теперь native Zed donut.
- `Model` selector: в hover для `Speed: Fast/Flex` добавить пояснение, что это request для новых turns when available, а не жесткая гарантия service tier.
- `Permissions` selector: сделать hover еще плотнее и предсказуемее: `Workflow: Plan` + `Permissions: Workspace/Read only/...`, чтобы было ясно, что Plan не стирает сохраненный sandbox/approval profile.
- `Compact now`: при запуске compaction держать в selector короткое, заметное состояние, например `Compacting...`; если позже появится дешевый progress/stale usage hint, можно заменить на более информативный compact marker.
- `Follow`: продолжать улучшать `ToolCallLocation` там, где app-server дает точные пути. Для search/list command actions использовать директорию поиска, включая `"." -> cwd`, а для shell fallback оставлять только очевидные одиночные read/write paths.

Already applied from this backlog:

- Command approval cards no longer use generic `Details` when the app-server asks for command permission. The adapter now reuses command-title heuristics for approval cards, including `Network access: host` for `curl`/`wget`/`iwr` style commands and `Create file: name` / `Create folder: name` / `Delete path: name` for obvious file operations. This matters because Zed collapses the approval body after completion and leaves the title as the durable visible text.
- Plain shell command approvals now pre-register the command as a display-only terminal before `request_permission`, so current Zed uses its terminal-card renderer for the pending approval. This is adapter-side only for shell/unknown commands; richer Allow/Deny + granularity dropdown grouping still belongs to Zed's internal `PermissionOptions::Dropdown` path.
- Shell-backed file/fetch operations now keep Zed-like tool kinds and summaries (`Fetch`, `Copy`, `Move`, `Delete`, `Create directory`) instead of being flattened into generic terminal cards.
- `Reasoning effort` now lives inside the grouped `Model` selector as the `Effort` section. Effort labels/descriptions are adapter-config preferences, so the selector keeps only operational model/effort/speed choices.
- Selector preferences are now JSONC-friendly and sectioned: `display`, `defaults`, `model_selector`, `layout`, `slash_commands`. `model_selector` covers defaults and clutter control that current Zed cannot persist for nested options; `models`, `reasoning_efforts`, and `slash_commands` are ordered lists, so commenting out a row hides it and row order controls menu order. `models` accepts either string ids or `{ id, name, description }` objects, and `reasoning_efforts` accepts effort ids or `{ id, name, description }` objects, so users can override labels and hover text for local translation. Selecting model/effort/speed through the grouped `Model` selector writes it back as the adapter default. Explicitly enabled `none` / `minimal` reasoning efforts can be surfaced for experiments even when the backend model list does not advertise them; unsupported combinations may still fail at backend turn start. Live check: `none` can answer, while `minimal` currently fails when app-server enables `image_gen` / `web_search` because those tools are incompatible with `reasoning.effort = "minimal"`.
- `/status` now uses the same compact section model as selector hover: `Context`, `Limits`, `Session`, `Integrations`, while preserving MCP/skills/plugins reports.
- Zed `1.3.0` / `zed-industries/zed#56446` supports editing external-agent thread titles in the client. Current Zed stores that edit as sidebar `title_override`; backend-persistent Codex names still go through adapter-side `/rename`.

## 4.2 Текущие Ограничения Zed UI

Ниже то, что уже проверено на практике и не стоит пытаться "чинить" только силами адаптера:

- Для ACP tool call у command approval `title` обязателен. Полностью убрать вторую строку заголовка нельзя без client-side изменений в `Zed` или без сомнительных zero-width hacks.
- `Zed` агрессивно обрезает длинный `title`, поэтому полный `reason` лучше держать в body, а не в заголовке карточки.
- Внутренний padding и layout approval-card задает сам `Zed`; адаптер может только слегка подправлять текстовое содержимое, но не контролирует нативные отступы контейнера.
- Если разбивать approval body на несколько ACP content-item'ов, `Zed` рисует между ними разделители. Для code block это выглядит плохо, поэтому command approval body у нас теперь отдается одним markdown-блоком.
- Верхний label `Run Command` тоже рисует сам клиент по `ToolKind::Execute`; адаптер может менять только вторичный title и содержимое body.
- Набор кнопок в command approval popup задается adapter-side ACP options. Форк теперь маппит их из `available_decisions` app-server, но точные подписи и группировка все еще ограничены тем, как `Zed` рисует permission options.
- Компактный Allow/Deny + granularity dropdown — это внутренний `Zed` `PermissionOptions::Dropdown` / `DropdownWithPatterns`, а не сериализуемая ACP option schema. Внешний ACP server bridge в текущем Zed заводит `request_permission` как `PermissionOptions::Flat(args.options)`, поэтому adapter-side можно только дать terminal-card preflight для plain shell approval, но не включить dropdown grouping без Zed-side изменения.
- Правый красный `x` у failed/cancelled tool/permission cards — client-side status indicator в `Zed`. Адаптер передает ACP status, но не управляет самой иконкой и не может adapter-side добавить симметричную зеленую галочку для completed state. Практичный adapter-side путь — не маппить успешные/обычные исходы в failed/cancelled без причины и держать title/content достаточно информативными после сворачивания карточки.
- Selected-agent / `New Thread` trigger в текущем `Zed` может визуально пульсировать только пока движется указатель мыши. По фактическому поведению это больше похоже на client-side repaint/animation quirk, чем на отдельную задержку ACP startup path.
- Нативный retry/error banner над composer, как у встроенного Zed Agent (`error sending request... Retrying...`), сейчас не является ACP surface для external agents. Адаптер может показать status/error в чате, но не может отрисовать этот жёлтый transient banner без Zed-side патча или нового ACP/Zed контракта.
- Нативный left-toolbar `Thinking Mode` / effort split-button с иконкой, как у встроенного Zed Agent, сейчас рендерится только для native thread. ACP `SessionConfigOptionCategory::ThoughtLevel` может служить подсказкой для keyboard/icons/placement, но текущий external `ConfigOptionsView` рисует такие options обычными text selectors справа. Adapter-side можно дать `Reasoning`/`Thinking` selector и hover, но иконка, split-button и placement слева требуют Zed-side поддержки.
- Нативный context circle для custom ACP CAS идет через ACP `usage_update` (`used` + `size`). Adapter-side отправляет этот update на live usage и на load/resume из cached usage; после текущего обновления Zed это уже видно как круговой context donut в toolbar для нашего внешнего агента. Текстовый `Status` selector и `/status` больше не дублируют context usage и остаются для account limits, session metadata, integrations and compaction action.
- Zed `External Agent settings` в свежих версиях показывает running agent version из ACP `agent_info`. Форк уже передает `codex-acp-cas` + `CARGO_PKG_VERSION`; если в UI версия не видна, это надо проверять как client-side/settings-path issue, а не добавлять отдельный chat notice.
- Нативная toolbar-кнопка Fast Mode в свежем `Zed` завязана на native-thread/staff/model gating и `supports_fast_mode()`. Для custom ACP CAS она не является универсальным внешним контролом, поэтому форк поднимает отдельный `Speed` selector через обычный ACP session config path без патча клиента.
- Для ACP session history `updated_at` реально показывается только в полном history-view и во встроенном блоке `Recent` внутри чата. Toolbar/dropdown `Recently Updated` в `Zed` рендерит только `title`, без времени, `cwd` и `meta`.
- `cwd` и `meta` в `AgentSessionInfo` до клиента доезжают, но текущие history/render пути `Zed` их не рисуют. Это уже client-side ограничение, а не баг адаптера.

## 5.1 Рабочий Процесс `mini` Ветки

`main` остается стабильной веткой для проверенных изменений, release-flow и тегов
`v*`. Ветка `mini` используется как экспериментальный буфер для работы на более
дешевой/слабой модели, чтобы не тратить лишний лимит `GPT-5.5 High` и не рисковать
релизной веткой при проверке гипотез.

Правила работы:

- Новые небольшие UX-гипотезы, docs-заметки и безопасные эксперименты можно вести
  в `mini`.
- `mini` не тегируется `v*` и не считается релизной веткой.
- Перед переносом изменений из `mini` в `main` нужно просмотреть diff и прогнать
  обязательные проверки: `cargo fmt --all -- --check`, `cargo test`,
  `cargo clippy --all-targets --all-features -- -D warnings`.
- Для user-facing изменений перед релизным переносом дополнительно запускать
  `bash script/build_local_release.sh`.
- Если слабая модель сделала плохой diff, чинить или переписывать нужно `mini`, не
  трогая стабильный `main`.
- Переносить удачные изменения в `main` через merge или cherry-pick после review.
  Если change-set user-facing, вместе с переносом обновлять версию по политике
  релизов.
- Релизный порядок: `main` -> approved changes from `mini` -> version/checks/local
  release build -> commit/push -> tag `v*` только из `main`.

## 6. Пока Можно Спокойно Не Трогать

- `close_session`: в текущем `Zed` практическая ценность низкая, пока клиент сам не умеет закрывать ACP-сессию и сразу открывать новую для clean sidebar.
- `fs/watch`
- override feature flags
- `codex_home` из `initialize`
- remote auth through client
- Нативный Zed toolbar Fast Mode button для custom ACP CAS: текущий практичный путь уже закрыт adapter-side `Speed` selector, а полное toolbar parity требует client-side UX контракта.
- Нативный Zed `Thinking Mode` / effort split-button для custom ACP CAS: без client-side mapping `ThoughtLevel` config options в left-toolbar контрол это останется обычным external selector, поэтому не добавлять adapter-side "готовую" иконку/placement-фичу.
- `DynamicToolCall`: потенциал есть как у моста к client-side/native UX, но для текущего Zed нет достаточно сильной surfaced-поверхности, чтобы держать даже partial runtime-support в основном коде. Возвращаться к нему имеет смысл только если появится конкретный Zed-side use case: например client-native picker, structured editor context или другой интерактивный UI, который нельзя нормально закрыть текущими ACP primitives. Для такого возврата сохранен backup-конспект в `docs/drafts/dynamic-tool-call-backup.md`.
